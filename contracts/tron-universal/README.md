# Tron Universal Contracts

Tron-compatible forks of contracts that cannot compile with Tron's solc (0.8.25). These exist solely to work around Tron solc limitations. In practice, only `SP1HeliosTron` appears strictly necessary today; `SpokePoolTron` and `Universal_SpokePoolTron` can likely be avoided with a small upstream-safe change to `SpokePool.sol`.

## Why these forks exist

Tron's Solidity compiler supports up to **solc 0.8.25**. Two incompatibilities prevent the original contracts from compiling:

### 1. SP1HeliosTron

**Original:** `contracts/sp1-helios/SP1Helios.sol` (pragma `^0.8.30`)

**Issue:** `require` with custom errors (e.g. `require(condition, CustomError(arg))`) was introduced in **Solidity 0.8.26**. SP1Helios uses this syntax in 7 places.

**Fix:** Replace all `require(condition, CustomError(...))` with `if (!condition) revert CustomError(...)`. This is semantically identical â€” both revert with the same custom error selector and arguments.

### 2. SpokePoolTron / Universal_SpokePoolTron

**Original:** `contracts/SpokePool.sol` and `contracts/Universal_SpokePool.sol`

**Issue:** `SpokePool.sol` has two `using ... for address` directives in scope that both define `isContract`:

- `using AddressLibUpgradeable for address` (line 46 of SpokePool.sol)
- `using AddressUpgradeable for address` (line 20 of SafeERC20Upgradeable.sol, pulled in via `using SafeERC20Upgradeable for IERC20Upgradeable`)

On solc >=0.8.26, the compiler resolves this correctly since both functions have identical signatures. On solc 0.8.25, the compiler treats this as an ambiguity error: `Member "isContract" not unique after argument-dependent lookup in address`.

**Current fork-based fix:** `SpokePoolTron` is a full copy of `SpokePool.sol` with the two `addr.isContract()` calls replaced by explicit `AddressLibUpgradeable.isContract(addr)`. `Universal_SpokePoolTron` then inherits from `SpokePoolTron` instead of `SpokePool`.

**Why the override approach does not work:** A simpler approach would be to mark the affected functions as `virtual` in `SpokePool.sol` and override them in `SpokePoolTron`. However, Solidity compiles the entire inheritance tree â€” the compiler parses and type-checks every function body in base contracts, even ones that are overridden at runtime. So even with overrides, Tron's solc 0.8.25 still encounters the ambiguous `address(to).isContract()` in `SpokePool.sol` and fails before it ever reaches the overriding implementation.

**Lower-maintenance alternative:** the ambiguity can be removed directly in `contracts/SpokePool.sol` with a two-line source change that is behavior-preserving on all supported compilers:

1. In `_unwrapwrappedNativeTokenTo`:

```solidity
// current
if (!address(to).isContract() || _is7702DelegatedWallet(to)) {

// alternative
if (!AddressLibUpgradeable.isContract(address(to)) || _is7702DelegatedWallet(to)) {
```

This is currently at `contracts/SpokePool.sol:1548`.

2. In `_fillRelayV3`:

```solidity
// current
if (updatedMessage.length > 0 && recipientToSend.isContract()) {

// alternative
if (updatedMessage.length > 0 && AddressLibUpgradeable.isContract(recipientToSend)) {
```

This is currently at `contracts/SpokePool.sol:1670`.

With those two edits made in the shared upstream file:

- `contracts/tron-universal/SpokePoolTron.sol` would no longer be necessary.
- `contracts/tron-universal/Universal_SpokePoolTron.sol` would no longer be necessary.
- `contracts/Universal_SpokePool.sol` could continue to inherit from `SpokePool` unchanged.

In other words, the compiler issue is real, but the full `SpokePoolTron` copy is not the only workaround. It is only required if we choose not to touch the shared `SpokePool.sol` source.

**Why not use standard solc?** The original `SpokePool.sol` compiles fine with standard solc 0.8.30 targeting `evm_version = "cancun"`, which would also eliminate the ambiguity issue. However, we use Tron's custom solc (`bin/solc-tron`) to ensure the generated bytecode is fully compatible with Tron's TVM. Until we can verify that standard solc bytecode deploys and executes correctly on Tron, keeping compatibility with `bin/solc-tron` remains the safer approach.

## Foundry profile

These contracts are compiled with the `tron-universal` Foundry profile:

```bash
FOUNDRY_PROFILE=tron-universal forge build
```

This profile uses `bin/solc-tron` (solc 0.8.25) and outputs artifacts to `out-tron-universal/`.

## Recommendation

If we optimize for minimum divergence from upstream contracts:

- Keep `contracts/tron-universal/SP1HeliosTron.sol`.
- Replace the two ambiguous `.isContract()` calls directly in `contracts/SpokePool.sol`.
- Delete `contracts/tron-universal/SpokePoolTron.sol`.
- Delete `contracts/tron-universal/Universal_SpokePoolTron.sol`.

If we instead optimize for leaving shared contracts untouched, the current fork-based setup is valid, but it carries ongoing sync cost because `SpokePoolTron` duplicates nearly the entire `SpokePool.sol` implementation.

## Keeping forks in sync

If the current fork-based approach is retained, then when `SP1Helios.sol`, `SpokePool.sol`, or `Universal_SpokePool.sol` are updated, the corresponding Tron fork must be updated to match. The forks are intentionally minimal in logic, but `SpokePoolTron` and `Universal_SpokePoolTron` still duplicate large amounts of source and therefore increase maintenance burden relative to the two-line upstream `SpokePool.sol` alternative described above.
